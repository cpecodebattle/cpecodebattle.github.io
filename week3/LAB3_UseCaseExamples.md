## UC-01: View Activity History

Primary Actor: Student
Goal: ดูรายการกิจกรรมทั้งหมดที่เคยเข้าร่วมและชั่วโมงกิจกรรมที่ได้รับ

### Preconditions
- Student ต้องเข้าสู่ระบบ (Authenticated) สำเร็จแล้ว
- ระบบมีข้อมูลกิจกรรมที่ Student เคยเข้าร่วมอย่างน้อย 1 กิจกรรม

### Postconditions
- Student เห็นกิจกรรมที่เคยเข้าร่วมและชั่วโมงกิจกรรมที่ได้รับ พร้อมชั่วโมงสะสมรวม

### Main Flow (Happy Path)
1. Student เลือกเมนูหรือกดปุ่ม "ประวัติกิจกรรม / History"
2. ระบบดึงข้อมูลกิจกรรมทั้งหมดที่ Student เข้าร่วมสำเร็จแล้ว
3. ระบบแสดงตารางหรือรายการกิจกรรม พร้อมรายละเอียด: ชื่อกิจกรรม, วันที่เข้าร่วม, ชั่วโมงกิจกรรมที่ได้รับ
4. ระบบคำนวณและแสดง ชั่วโมงกิจกรรมสะสมรวม ท้ายตาราง
5. Use Case จบลง

### Alternative Flows / Exceptions
- AF-1 – ไม่มีประวัติการเข้าร่วม**
  1. ที่ขั้นตอน 2 ระบบไม่พบข้อมูลกิจกรรมที่ Student เคยเข้าร่วม
  2. ระบบแสดงข้อความ **“คุณยังไม่เคยเข้าร่วมกิจกรรมใด ๆ”**
  3. Use Case จบลง







## UC-02: Propose New Activity

Primary Actor: Member (Student / Staff)
Goal: เสนอแบบฟอร์มขอจัดกิจกรรมเพื่อรอการอนุมัติจากผู้มีอำนาจ

### Preconditions
- Member ต้องเข้าสู่ระบบ (Authenticated ด้วยบัญชีมหาวิทยาลัย)

### Postconditions
- ระบบบันทึกข้อเสนอจัดกิจกรรมพร้อมสถานะ "Pending Approval"

### Main Flow (Happy Path)
1. Member เลือกเมนูหรือกดปุ่ม **"เสนอจัดกิจกรรม / Access Event"**
2. ระบบแสดงแบบฟอร์มให้กรอกรายละเอียดข้อเสนอ (ชื่อ, วันที่, เป้าหมาย, ผู้เข้าร่วมที่คาดหวัง, ฯลฯ)
3. Member กรอกข้อมูลที่จำเป็นทั้งหมด
4. Member กดปุ่ม **"Submit Proposal"**
5. ระบบตรวจสอบความสมบูรณ์ของข้อมูล และบันทึกข้อเสนอ
6. ระบบแสดงข้อความยืนยัน “เสนอกิจกรรมสำเร็จ รอการอนุมัติ”
7. Use Case จบลง (รอ Primary Actor ใน UC-03 มาดำเนินการต่อ)

### Alternative Flows / Exceptions
- AF-1 – ข้อมูลฟอร์มไม่สมบูรณ์**
  1. ที่ขั้นตอน 5, Member กรอกข้อมูลที่จำเป็น (Required Field) ไม่ครบถ้วน
  2. ระบบแจ้งเตือนให้กรอกข้อมูลให้ครบถ้วน โดยเน้น Field ที่ขาดหายไป
  3. กลับไปขั้นตอนที่ 3 เพื่อให้ Member แก้ไขข้อมูล






## UC-03: Approve Activity Proposal

Primary Actor: Approver (University Official / Activity Chairman)
Goal: ตรวจสอบข้อเสนอจัดกิจกรรมที่ถูกส่งเข้ามา และอนุมัติ/ปฏิเสธข้อเสนอนั้น

### Preconditions
- Approver ต้องเข้าสู่ระบบ (Authenticated) และมีสิทธิ์ในการอนุมัติกิจกรรม
- มีข้อเสนอจัดกิจกรรมที่สถานะ "Pending Approval" อย่างน้อย 1 รายการ

### Postconditions
- สถานะกิจกรรมเปลี่ยนเป็น "Approved" หรือ "Rejected"
- ระบบบันทึกการตัดสินใจพร้อมเหตุผล/Feedback (ถ้ามี)

### Main Flow
1. Approver เลือกเมนู "ตรวจสอบข้อเสนอ / Event Submission"
2. ระบบแสดงรายการข้อเสนอที่สถานะ "Pending Approval"
3. Approver เลือกข้อเสนอที่ต้องการและดูรายละเอียดทั้งหมด
4. Approver เลือกปุ่ม "Approve" (หรือ "Pass")
5. ระบบบันทึกการอนุมัติและอัปเดตสถานะกิจกรรมเป็น "Approved"
6. ระบบแจ้งเตือน/ส่งอีเมลไปยังผู้เสนอ
7. Use Case จบลง

### Alternative Flows / Exceptions
- AF-1 – ปฏิเสธพร้อม Feedback**
  1. ที่ขั้นตอน 4, Approver เลือกปุ่ม "Reject" (หรือ "Not Pass")
  2. ระบบบังคับให้ Approver กรอก Feedback/เหตุผล ในการปฏิเสธ
  3. ระบบบันทึกการตัดสินใจและอัปเดตสถานะเป็น "Rejected"
  4. ระบบแจ้งเตือนพร้อม Feedback ไปยังผู้เสนอ
  5. Use Case จบลง
- **AF-2 – ไม่มีสิทธิ์ในการอนุมัติ**
  1. ที่ขั้นตอน 1, ผู้ใช้เข้าถึงเมนู "ตรวจสอบข้อเสนอ" แต่ระบบพบว่าไม่มีสิทธิ์ "Approver"
  2. ระบบปฏิเสธการเข้าถึงและแสดงข้อความ "คุณไม่มีสิทธิ์ในการอนุมัติกิจกรรม"
  3. Use Case จบลง
